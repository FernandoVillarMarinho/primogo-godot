1 Abrir o jogo em tela cheia fácil · 5 min
project.godot
🐛 O problema No computador o jogo abria num quadradinho minúsculo. O projeto declara a "tela virtual" de 720×1280 (celular em pé), mas nunca dizia ao Windows como abrir a janela.
🧭 O conceito: janela ≠ tela virtual No Godot, o jogo é desenhado numa tela virtual (viewport) e depois esticado para a janela real. Três configurações controlam isso: o modo da janela (janela/maximizada/tela cheia), o modo de esticar (stretch mode) e a proporção (aspect). Com aspect=keep, o jogo nunca distorce: sobram barras nas laterais, como um filme de cinema na TV.

Passo a passo

  1. No editor: Projeto → Configurações do Projeto → Display → Window.
  2. Em Size, mude Mode para Fullscreen.
  3. Preencha Window Width/Height Override com 486 × 864 — é o tamanho da janela caso alguém saia da tela cheia (mesma proporção do 720×1280).
  4. Em Stretch, confirme Mode = canvas_items e Aspect = keep.
[display]

window/size/viewport_width=720
window/size/viewport_height=1280
window/stretch/mode="canvas_items"
window/handheld/orientation=1
[display]

window/size/viewport_width=720
window/size/viewport_height=1280
window/size/mode=3                     # 3 = tela cheia
window/size/window_width_override=486   # janela ampla, se sair da tela cheia
window/size/window_height_override=864
window/size/initial_position_type=1     # centrada no monitor
window/stretch/mode="canvas_items"
window/stretch/aspect="keep"            # nunca distorce: barras, não esticado
window/handheld/orientation=1

2 O mistério do 142 (que era 143) média · 25 min
domain/levels/level_data.gd domain/tests/test_level_data.gd
🐛 O problema A fase 1-6 mostrava 142 num bloco de gelo — mas 142 não é divisível por 5 nem por 13, então a fase travava. O número desenhado pela equipe era 143 (11 × 13). E não era caso único: 63 números em 24 fases estavam corrompidos (uma fase mostrava até 0).
🧭 O conceito: computadores erram frações Cada fase guarda um "disfarce" r que multiplica o valor real: na 1-6, o real é 55 e r = 2.6, logo 55 × 2.6 = 143. Só que o computador não consegue guardar 2.6 exato — guarda 2.5999999… Então 55 × 2.5999999 = 142.99999, e a conversão para inteiro do jogo antigo truncava (cortava a parte decimal): virava 142. A correção: se o resultado está coladinho num inteiro (a menos de 0,01), arredonde; senão, trunque como antes.

Passo a passo

  1. Abra domain/levels/level_data.gd no editor de script (Ctrl+Shift+O busca arquivos pelo nome).
  2. Ache em generate_frozen() a linha que calcula o valor exibido.
  3. Extraia a conta para uma função nova _displayed_for() com a tolerância.
  4. Escreva o teste ANTES de dar por pronto: ele fixa o caso real da fase 1-6.
# dentro de generate_frozen():
var displayed_value: int = int(true_value * rp)  # trunca, como o (int) do C#
# dentro de generate_frozen():
var displayed_value := _displayed_for(true_value, rp)

## r é sempre uma razão desenhada para dar produto INTEIRO (ex.: 2.6 = 13/5), mas
## 2.6 não existe em float32 (vira 2.5999999...) e 55 × 2.5999999 = 142.99999 → o
## truncamento puro exibia 142 num tabuleiro em que só 143 é divisível. Produto a
## menos de 0,01 de um inteiro arredonda; fora disso, truncamento legado.
static func _displayed_for(true_value: int, rp: float) -> int:
	var product := true_value * rp
	if absf(product - roundf(product)) < 0.01:
		return roundi(product)
	return int(product)

E o teste que prova (em test_level_data.gd):

func test_displayed_rounds_float32_error_level_01_06() -> void:
	var els := [_elem(2, 2, 13), _elem(0, 4, 5), _elem(4, 0, 11)]

	assert_eq(lvl.generate_frozen()[1]["displayed_value"], 143,

4 A festinha da conquista (tweens e arcos) avançada · 45 min
features/board/board.gd
🐛 O problema A conquista de um primo merecia mais emoção — e, na última conquista da fase, a tela de vitória atropelava a animação.
🧭 O conceito: Tween, a linha do tempo de bolso Um Tween anima propriedades ao longo do tempo: tween_property(nó, "scale", Vector2.ONE, 0.22) leva a escala até 1 em 0,22 s. Encadeando chamadas você monta uma coreografia: pulo → espera → voo → encaixe. Para o voo em arco, usamos uma curva de Bézier quadrática: a posição desliza de A para B passando perto de um ponto de controle lá em cima — matemática de 1 linha dentro de um tween_method.

E a regra de ouro do item 6.1: a vitória espera o tween acabar de verdade (await tween.finished), nunca um cronômetro chutado.

Passo a passo (o esqueleto da coreografia)

  1. Monte um holder com a chama animada atrás e o número na frente (a ordem dos filhos define quem desenha por cima).
  2. Pulo de entrada: escala 0.3 → 1.0 com TRANS_BACK (passa do ponto e volta — o "quique" de desenho animado).
  3. Voo em arco até o slot correto da fileira ordenada.
  4. No pouso: some com o número voador e pulse o slot de destino (por isso a lição 3 guardou _slot_nodes).
  5. Guarde o tween em _conquest_tween — quem mostra a vitória faz await _conquest_tween.finished antes do modal.
# o voo em ARCO: Bézier quadrática dentro de um tween_method
var apex := Vector2((start.x + target.x) / 2.0, minf(start.y, target.y) - 160.0)
tw.tween_method(func(t: float) -> void:
	holder.position = start.lerp(apex, t).lerp(apex.lerp(target, t), t),
	0.0, 1.0, 0.5).set_trans(Tween.TRANS_QUAD).set_ease(Tween.EASE_IN)

tw.tween_callback(func() -> void:   # o pouso confirma o encaixe
	holder.queue_free()
	AudioBus.play_effect(AudioBus.SFX_PRIME_SWAP)
	_pulse_slot(value))
tw.tween_interval(0.28)      # respiro do pulso antes de qualquer modal
_conquest_tween = tw         # a tela de vitória ESPERA este tween

# ... e onde o fim de fase é mostrado:
if _conquest_tween != null and _conquest_tween.is_running():
	await _conquest_tween.finished   # espera REAL, não timer chutado

5 O tutorial que não descongelava o 6 avançada · 45 min
features/tutorial/tutorial_sequence.gd features/tutorial/tutorial_overlay.gd resources/levels/level_02_01.tres
🐛 O problema Na fase 2-1 (tutorial), mover o 3 e tentar descongelar o 6 não funcionava — nunca.
🧭 O conceito: depure no PAPEL antes de culpar o código Desenhe a grade 5×5: fogo em (2,2), o 6 em (4,3), o 4 em (0,1). O roteiro antigo era DIREITA → ESQUERDA. Siga com o dedo: DIREITA leva o fogo até (4,2) — linha 2. ESQUERDA volta pela mesma linha 2. O 6 está na linha 3. O fogo passa por cima, por baixo, dá até tchau — mas nunca TOCA o 6. Não era colisão, não era clique: era um roteiro geometricamente impossível. A correção: DIREITA → BAIXO (de (4,2) o 6 está logo abaixo!) → clique no primo 2 → ESQUERDA — e o 4 muda para (0,3), na rota da ESQUERDA final.

Passo a passo

  1. Em tutorial_sequence.gd, troque a sequência: T2 := [RIGHT, DOWN, BALLOON, LEFT].
  2. Em level_02_01.tres (pode abrir como texto!), no elemento do 4, troque y = 1 por y = 3.
  3. Crie as instruções por passo — um Array de frases paralelo à sequência (captions_for()) — e um cartaz (PanelContainer + Label) ancorado no topo que mostra a frase do passo atual.
  4. No passo do balão, faça o slot do primo 2 pulsar e a mãozinha apontar para ele (o board chama _tutorial.point_to(posição_do_slot)).

6 Fases em ordem de leitura (1, 2, 3 →) fácil · 15 min
features/level_select/level_grid.gd domain/tests/test_level_select.gd
🐛 O problema A primeira LINHA mostrava 1, 5, 9 (preenchimento por colunas). Criança lê como livro: esquerda → direita, cima → baixo.
🧭 O conceito: uma fórmula, doze botões A grade 3×4 numera cada posição com UMA fórmula. Por colunas: col×4+linha (1,5,9 na primeira linha). Por linhas: linha×3+col (1,2,3). Como todos os botões nascem da mesma fórmula, trocar a fórmula troca a grade INTEIRA — e o botão continua abrindo a fase certa, porque o número mostrado e a fase aberta vêm do mesmo cálculo. Nunca conserte grade botão por botão.
static func level_index(col: int, row: int) -> int:
	return col * ROWS + row   # por colunas: 1,5,9 / 2,6,10 ...
static func level_index(col: int, row: int) -> int:
	return row * COLS + col   # por linhas: 1,2,3 / 4,5,6 ... (ordem de leitura)

7 Créditos: pular, destravar o Jogar e arrumar a composição avançada · 50 min
features/main_menu/credits.gd features/main_menu/main_menu.gd
🐛 Três problemas de uma vez (a) não dava para sair dos créditos; (b) depois deles, o botão Jogar "morria"; (c) a logo do DJDE cobria a do Primogo e os nomes da Idealização estavam sem sombra.
🧭 O conceito: o fantasma invisível que rouba cliques O bug (b) é um clássico de Godot: um Control de tela cheia com mouse_filter = STOP (o padrão!) intercepta TODOS os cliques — mesmo transparente. Os créditos terminavam, ficavam invisíveis… e continuavam lá, engolindo os cliques no Jogar. O botão nunca esteve desativado: havia um vidro na frente dele. Regra prática: overlay que terminou é overlay escondido (visible = false), e todo elemento decorativo leva mouse_filter = IGNORE.

Para o "Pular" (a), o desafio é abortar uma apresentação que está no meio de um await: usamos um número de rodada (_run_id) — o pulo incrementa o número e adianta o tween até o fim; quando o play() acorda, vê que a rodada mudou e desiste em silêncio.

Passo a passo

  1. Transforme os 3 painéis de PNG em painéis de texto vivo (um construtor _build_text_panel() que recebe título + seções) — assim TODOS os nomes ganham a mesma sombra, fonte e centralização (resolve o (c) dos nomes).
  2. Ancore a logo do DJDE embaixo: set_anchors_preset(Control.PRESET_BOTTOM_WIDE) com offsets negativos — ela cola no jardim verde em QUALQUER resolução (nada de posição fixa).
  3. Adicione o botão Voltar (mesma arte da seleção) que chama skip().
  4. No fim (normal ou pulado): visible = false + sinal credits_finished → o menu religa os botões.
func play() -> void:
	visible = true
	_run_id += 1
	var id := _run_id                    # o número desta rodada
	for i in _panels.size():
		await _run_segment(_panels[i], hold_de(i))
		if id != _run_id:
			return                       # alguém pulou: esta rodada morreu
	_finish()

func skip() -> void:                     # item 2.1: sair a qualquer momento
	_run_id += 1                         # invalida a rodada em andamento…
	if _tween != null and _tween.is_valid():
		_tween.custom_step(999.0)        # …e adianta o tween até o fim (acorda o await)
	_finish()

func _finish() -> void:
	visible = false                      # o "vidro" some: Jogar volta a receber cliques
	credits_finished.emit()

Glossário de bolso

TermoTradução de professor
Cena / NóTudo no Godot é um nó (um botão, uma imagem, um som); uma cena é uma árvore de nós salva num arquivo.
Sinal (signal)Um grito que um nó dá ("fui clicado!", "a partida acabou!") e outros nós escutam. É como as telas do Primogo conversam sem se conhecerem.
TweenLinha do tempo de bolso: leva uma propriedade (posição, escala, cor) de A a B em X segundos, com estilo (quique, mola, suave).
await"Espere este evento terminar antes de seguir." A lição 4 inteira gira nisso.
AutoloadNó que existe o jogo inteiro (o DJ do som, o porteiro das telas, o cartório do save).
.tresArquivo de dados legível (as fases do Primogo). Pode abrir no bloco de notas.
mouse_filterSe um Control engole (STOP), repassa (PASS) ou ignora (IGNORE) os cliques. O vilão da lição 7.
GUTO robô que roda os 148 testes: joga milhares de partidas em 3 segundos e acusa qualquer regra ferida.
float32Número com vírgula de precisão limitada — o culpado do 142. Computadores não sabem guardar 2.6.
HeadlessRodar o Godot sem janela, só no terminal — como os robôs de teste fazem.
